# 2026-03-25 GCSダウンロードタイムアウト

## 概要

**SEV**: SEV-3

Cloud Storageからの画像ダウンロードが断続的にタイムアウトし、処理が途中で失敗する問題が発生。実際のジョブでPage 2の処理がタイムアウトで失敗するケースが確認された。

## タイムライン

- 本番ジョブ（ffc24728）でPage 2のダウンロードがタイムアウトして失敗
- 最初の対応としてリトライロジック（3回、2秒間隔）とタイムアウト設定を追加
- それでも断続的にタイムアウトが発生
- ログ調査により`getMetadata()`の呼び出しが毎回21秒かかっていることを発見
- `getMetadata()`完了後に`file.download()`を呼ぶと既存のタイムアウト（120s）が残り少なくなり失敗していた
- `getMetadata()`を削除し、ダウンロード後に`fs.stat()`でファイルサイズを確認する方式に変更して解消

## 根本原因

`getMetadata()`がCloud Storage APIへの追加ラウンドトリップを発生させ、毎回21秒の遅延を引き起こしていた。これがダウンロードのタイムアウト余裕を食いつぶしていた。本来ファイルの存在確認目的だったが、ダウンロード後のファイルサイズ確認で代替できた。

## 影響範囲

- **重大度**: SEV-3
- **影響ユーザー数**: 複数（実ジョブで失敗が発生）
- **ダウンタイム**: 断続的な処理失敗（数日間）
- **TTD（検知まで）**: 実ジョブ失敗のログから即時
- **TTR（復旧まで）**: 約半日（根本原因の特定に時間を要した）
- **データ損失**: なし（リトライで再処理可能）
- **外部影響**: なし

## 解決策

`getMetadata()`の呼び出しを削除し、ダウンロード後の`fs.stat()`でファイル整合性を確認する方式に変更。

## 再発防止策

- [ ] GCS操作のレイテンシをログに記録し、異常に遅い呼び出しを検知できるようにする
- [ ] ファイル存在確認の方法として`getMetadata()`よりも軽量な手段を優先する
- [ ] タイムアウト値は「正常系の所要時間 × 余裕係数」で設定し、根拠をコメントに残す
