# 2026-03-25 動画生成API 統合トラブル連発

## 概要

**SEV**: SEV-3

動画生成APIの初期統合時、誤ったエンドポイント・認証方式・レスポンス形式・リージョン・ポーリングURL形式を使用し、動画生成が一切機能しない状態が続いた。複数の根本原因が重なり、修正が4回に及んだ。

## タイムライン

- 動画生成APIをGemini APIのエンドポイント（`generativelanguage.googleapis.com`）で実装し統合テスト開始
- 404・400エラーが継続発生
- Vertex AI（`aiplatform.googleapis.com`）が正しいエンドポイントと判明し切り替え。APIキー認証からADCに変更
- ポーリングURLにoperation IDのUUIDのみを使用していたため400エラー「Operation ID must be a Long」が発生
- フルのoperation name（`projects/.../operations/{id}`）が必要と判明し修正
- リクエストのリージョンを`asia-northeast1`で実装していたがVeo APIは`us-central1`のみ対応と判明
- レスポンス形式がGCS URIではなくbase64エンコードされた動画バイナリだったため「No video GCS URI in response」エラーが発生
- base64デコードに対応して解消

## 根本原因

公式ドキュメントが不十分で、正しいエンドポイント・リージョン制約・レスポンス形式がドキュメント上で明確でなかった。また`@google-cloud/vertexai` SDKはVeoに非対応（Gemini専用）であり、REST APIを直接叩く必要があったが初期実装でSDKを使用しようとしていた。

## 影響範囲

- **重大度**: SEV-3
- **影響ユーザー数**: 0人（開発中・未リリース機能）
- **ダウンタイム**: 約1日（開発環境のみ）
- **TTD（検知まで）**: 即時（テスト時にエラー確認）
- **TTR（復旧まで）**: 約1日（4回の修正を経て解消）
- **データ損失**: なし
- **外部影響**: なし

## 解決策

1. エンドポイントをVertex AI REST APIに切り替え、ADC認証に変更
2. ポーリングURLをfull operation nameで構築するよう修正
3. リージョンを`us-central1`にハードコード
4. レスポンスのbase64動画バイナリをデコードして保存するよう対応

## 再発防止策

- [ ] 新規外部APIの統合時はサンドボックスで最小限の動作確認スクリプトを先に書く
- [ ] リージョン制約などの非自明な制限はコード内にコメントで明記する（実施済み）
- [ ] 公式サンプルコードとレスポンス形式を統合前に必ず確認する
